This new spec should improve the product in four dimensions simultaneously:
1. strategy creation usability
2. backtesting and research workflow usability
3. quant-trading robustness and realism
4. API and domain model ergonomics

Important constraints:
- Preserve clean architecture boundaries
- Avoid breaking existing JSON payloads unless explicitly versioned
- Prefer additive, backwards-compatible design
- Every major change must include test strategy
- Keep implementation incremental and PR-friendly
- Distinguish between V5 scope and later roadmap items
- Do not propose a reckless rewrite; prefer adapters, facades, and staged migration
- Where scope is large, split into V5 core scope vs V5.1/V6 roadmap

Context from review:
The engine is already strong in architecture, engine correctness, metrics breadth, and research workflow breadth. The current system has a solid layered structure, a correct V2 event-processing order to reduce look-ahead bias, broad metrics, and advanced workflows such as walk-forward, Monte Carlo, parameter sweep, perturbation, stability, randomized OOS, regime segmentation, and benchmark comparison.

But there are important weaknesses:
- strategy creation is too code-centric
- there is no typed parameter schema discovery for strategies
- ScenarioConfig is too large and acts like a god object
- some workflow/API endpoints are not ergonomic and may ignore user-provided options
- progress reporting exists conceptually but long-running jobs need a better user journey
- long-only assumptions are too embedded
- cross-asset portfolio support is limited
- quant realism can be extended further
- usability for creating, validating, comparing, and iterating on strategies should improve

The spec should cover the following areas.

==================================================
AREA A — STRATEGY CREATION EXPERIENCE
==================================================

Goal:
Make it much easier to create, discover, validate, and iterate strategies.

This area must be treated as a first-class product redesign, not just a few form tweaks.
Design the new strategy creation flow as a guided research workflow rather than a raw configuration form.

The design should assume this product is used for iterative strategy research, where the user wants to:
- discover a strategy or template
- configure it safely
- run a quick sanity check
- inspect results
- iterate into deeper validation
- compare versions
- move a strategy through research maturity stages

Please define requirements/design/tasks for the following.

1. Strategy metadata and schema discovery
- Introduce a typed strategy metadata contract
- Expose strategy display name, description, family, supported asset classes/timeframes, version, tags, and development stage
- Expose a typed parameter schema for each strategy:
  - parameter name
  - type
  - default value
  - required/optional
  - min/max
  - enum choices
  - description/help text
  - sensitivity hint (low/medium/high risk of overfitting)
  - grouping/category
  - advanced/basic visibility flag
  - display order
- Prefer attribute-driven or descriptor-driven metadata, but keep domain boundaries clean
- Design so the metadata can be consumed by API, CLI, and Web UI consistently

2. Strategy template / builder support
- Add support for strategy templates or guided creation for common strategy families:
  - trend following
  - mean reversion
  - breakout
  - momentum
  - rotation
  - experimental/custom
- Define what should be configurable without writing code
- Distinguish between:
  - code-defined strategies
  - template-generated strategies
  - imported configs
  - future DSL/no-code strategies
- Support forking an existing strategy/version into a new draft
- Support presets per family, such as conservative / balanced / aggressive where relevant

3. Validation before run
- Add preflight validation for strategy configuration
- Catch missing parameters, invalid ranges, impossible combinations, incompatible timeframe/data selections, unrealistic risk settings, invalid execution window choices, and sealed test set conflicts before execution
- Validation output should be structured for API and UI use
- Validation should support severity levels:
  - blocking error
  - warning
  - recommendation
- Validation should be usable both inline while editing and as a final preflight review before running

4. Strategy comparison usability
- Improve side-by-side comparison of strategies and versions
- Show what changed:
  - parameters
  - execution assumptions
  - data window
  - risk settings
  - realism settings
  - research stage
  - hypothesis text
- Show resolved/effective values, not just raw stored values
- Highlight materially significant changes, not just literal diffs

5. Product UX architecture for the new strategy builder
Design a new multi-step builder, not a single long page.

The default design should be:
- a multi-step wizard
- progressive disclosure
- preset-first, custom-second
- an always-visible summary rail
- a strong post-create handoff into strategy research workflow

Use a two-pane layout:
- Left/main pane: current step content
- Right pane: “Research Summary” rail that updates live

The right summary rail should show:
- strategy name
- strategy family
- template/source
- data file / symbol
- timeframe
- execution window
- in-sample / out-of-sample / sealed test split
- key parameters
- risk preset
- realism preset
- estimated bar count
- validation state
- “ready for quick run” / “needs fixes” / “research-grade ready” status
- next recommended action

6. Required builder flow
Design the strategy creation flow around the following screens/steps.

Step 1 — Choose a starting point
- Template strategy
- Existing strategy version
- Blank advanced strategy
- Import config

This screen should also allow choosing a strategy family via cards.
Each family card should show:
- plain-language description
- common use case
- typical markets/timeframes
- typical failure mode
- beginner-friendly or advanced badge

Step 2 — Select data and execution window
The user chooses:
- validated data file or symbol
- timeframe
- date range / execution window
- in-sample / out-of-sample split
- optional sealed test set

This step should include a timeline visualization showing:
- full data range
- training window
- OOS window
- sealed test set

This step should show live diagnostics:
- estimated bar count
- insufficient data warning
- timeframe mismatch warning
- overlap or leakage warnings
- data validation issues
- likely underpowered study warning

Step 3 — Configure strategy logic and parameters
Do not render parameters as one flat unstructured list.

Group parameters into sections such as:
- Signal
- Entry
- Exit
- Risk
- Filters
- Execution assumptions

The parameter editor should support:
- Simple mode
- Advanced mode

Simple mode:
- only the most decision-relevant parameters
- fewer controls
- opinionated defaults

Advanced mode:
- full parameter set
- full schema visibility
- fine-grained overrides

Each parameter row should support:
- label
- type-specific input
- default value
- help text
- “why this matters”
- overfitting sensitivity hint
- inline validation
- reset to default

Step 4 — Select realism and risk profile
Do not expose raw engine-level realism settings first.

Start with presets:
- Fast idea check
- Standard backtest
- Conservative realistic
- Research-grade validation

Then allow expansion into detailed controls:
- fill mode
- slippage model
- commission model
- spread assumptions
- session rules
- liquidity/volume constraints
- position sizing
- exposure limits
- prop-firm constraints if applicable

This step should include a “what this preset changes” explanation panel.

Step 5 — Review and launch
This is a preflight review, not a dead-end confirmation screen.

Show:
- strategy identity
- hypothesis
- effective configuration
- data selection and execution window
- IS / OOS / sealed split
- key parameters
- realism and risk assumptions
- estimated runtime
- validation findings
- suggested next action

Allow launch actions:
- Run quick sanity test
- Run standard backtest
- Save draft without running

7. Required post-create handoff

The default Overview should show:
- latest run
- current development stage
- research checklist
- key warnings
- recommended next study
- shortcuts to compare versions, edit assumptions, and launch validation studies

AREA B — SCENARIO / CONFIGURATION ERGONOMICS
==================================================

Goal:
Reduce friction and ambiguity when configuring runs.

Please define requirements/design/tasks for:

1. Refactor configuration shape
- Break ScenarioConfig into smaller sub-objects while maintaining backward compatibility
- Example groups:
  - DataConfig
  - StrategyConfig
  - RiskConfig
  - ExecutionConfig
  - ResearchConfig
  - ValidationConfig
- Provide migration/adapter logic for old payloads

2. Explicit override rules
- Eliminate confusing precedence rules such as top-level values silently overridden by nested values
- Where precedence must exist, make it explicit and validated
- Return warnings when legacy payloads rely on deprecated precedence

3. Config presets
- Add reusable presets for:
  - fast idea check
  - standard backtest
  - realistic backtest
  - research-grade validation
- Presets should populate sensible defaults but remain editable

4. Explainability
- Add a generated “resolved config” view
- Users should be able to see the final effective config after defaults, overrides, and presets are applied

AREA C — API AND JOB WORKFLOW IMPROVEMENTS
==================================================

Goal:
Make long-running backtests and studies easier to use and integrate.

Please define requirements/design/tasks for:

1. Job-based execution
- Introduce async jobs for long-running runs/studies
- POST creates a job and returns a job id
- Users can poll or subscribe to status/progress/results
- Support cancellation
- Support partial result persistence where appropriate

2. Progress reporting
- Standardize progress events across:
  - single runs
  - walk-forward
  - monte carlo
  - parameter sweep
  - perturbation/stability studies
- Include percentage, stage, current item, total items, elapsed time, and warnings

3. Endpoint ergonomics
- Ensure sweep/MonteCarlo/walk-forward endpoints fully honor caller-supplied options
- Use explicit request DTOs
- Improve error responses and validation responses
- Ensure consistency across CLI, API, and UI entry points

4. Strategy/schema discovery endpoints
- Add endpoints to list:
  - strategies
  - strategy schemas
  - research workflows
  - config presets
  - supported realism models / fill modes / commission models / slippage models

AREA D — USER JOURNEY AND PRODUCT UX
==================================================

Goal:
Make the app easier to use for iterative research, not just one-off runs.

Please define requirements/design/tasks for:

1. Ideal user journey
Design around this flow:
- discover strategies
- pick a strategy/template
- select validated data
- configure parameters with help text and defaults
- run a quick sanity backtest
- review results
- launch deeper validation studies
- compare versions
- promote a strategy through development stages
- run final validation
- export/share results

2. Pain point reduction
Address these friction points:
- too many low-level choices up front
- unclear defaults
- difficulty knowing what to do next
- poor visibility into long-running jobs
- unclear distinction between exploratory and validation-grade testing
- too much manual comparison between runs

3. Guided workflows
Add recommendations and guardrails:
- “quick check” vs “research-grade” modes
- suggest next studies based on current maturity
- show warnings when the user appears to be over-optimizing
- surface missing robustness checks

4. Result usability
Improve result interpretation:
- highlight most decision-relevant metrics
- show realism assumptions prominently
- show failure reasons clearly
- compare in-sample vs out-of-sample performance clearly
- show whether a strategy passes a minimum research checklist

AREA E — QUANT / RESEARCH IMPROVEMENTS
==================================================

Goal:
Make the engine more credible from a serious quant trading perspective.

Please define requirements/design/tasks for:

1. Realism improvements
Evaluate and propose improvements for:
- slippage realism
- spread handling
- commissions/fees
- fill assumptions
- session constraints
- liquidity/volume constraints
- bar-resolution caveats
- gap handling

2. Overfitting defenses
Expand and systematize:
- parameter sensitivity analysis
- parameter stability
- walk-forward

- Monte Carlo
- trial counting

- minimum sample size / minimum bars to test
- sealed test set enforcement

3. Portfolio and cross-asset support
Design a path for:
- better multi-asset portfolio awareness

- correlation-aware constraints

The current system has long-only assumptions in key abstractions.
Design an extensibility path that:
- avoids a reckless breaking rewrite

- isolates long-only assumptions
- adds tests to protect current behaviour

5. Benchmarking and evaluation
Improve comparison with:
- benchmark strategies
- buy-and-hold
